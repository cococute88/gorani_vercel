## Historical Dividends To Calendar Events

Each dividend row is normalized to:

```txt
{ date: YYYY-MM-DD, amount: number }
```

The quote `date` is treated as the ex-dividend date.

For every historical dividend row, the provider creates:

- `ex_div` event on the dividend date
- `buy_by` event on the previous weekday before the dividend date

The simple previous weekday helper only skips Saturday and Sunday. It does not implement a full exchange holiday calendar in this step.

Payment events are not created because `/api/quote/dividends` currently does not provide payment dates.

## Frequency Inference

`inferDividendFrequency` sorts historical dividend dates and calculates the median interval in days.

The first-pass mapping is:

| Median interval | Frequency | Month step |
| --- | --- | --- |
| 20-45 days | monthly | 1 |
| 70-110 days | quarterly | 3 |
| 150-220 days | semiannual | 6 |
| 300-430 days | annual | 12 |

If fewer than 3 historical dividend dates are available, projection is skipped and a warning is recorded.

If the median interval is outside those bands, projection is skipped and a warning is recorded.

## Future Estimated Events

`projectEstimatedDividendEvents` starts from the latest historical dividend date and advances by the inferred month step until it reaches the next future date.

Projection range:

- from today
- through the next 12 months

For each projected dividend date, the provider creates:

- estimated `buy_by` event
- estimated `ex_div` event

Projected dates that land on a weekend are moved to the next weekday. The buy deadline is then set to the previous weekday.

Projected events use:

```ts
status: "estimated"
sourceKind: "estimated"
```

The provider result warnings include a note when estimated events are generated.

## canonicalEventId And legacyEventId

All real generated events keep the Step 5A-1 identity contract:

- `id` is the canonical event id.
- `canonicalEventId` is set to the same canonical event id.
- `legacyEventId` preserves the old mock-style id shape: `{TICKER}-{UI_TYPE}-{DATE}`.

Examples:

```txt
dividend:SCHD:ex_div:2026-03-25
dividend:SCHD:buy:2026-03-24
```

Changing a provider row from estimated to declared does not change the canonical id as long as ticker, normalized event type, and event date stay the same.

## Cache Read/Write Priority

The real provider uses the per-ticker localStorage cache helpers from `lib/calendar-cache.ts`.

Read/write priority:

1. Fresh per-ticker localStorage cache
2. `/api/quote/dividends` real provider fetch
3. Stale per-ticker cache fallback if the provider fails or returns no dividends
4. Mock calendar fallback

Successful provider results are written with `saveCalendarTickerCache`.

Cache entries contain generated events only. User meta such as memo, star, and heart remains outside the cache in the existing event meta storage.

Firestore cache wrappers remain available from Step 5A-2, but this step does not force a Firestore cache connection.

## Fallback Order

Per ticker fallback order:

1. Use fresh cache if valid.
2. Fetch `/api/quote/dividends?ticker={TICKER}&range=5y`.
3. If dividends exist, build historical events and possible projected events, then save cache.
4. If dividends are empty and stale cache exists, use stale cache with warnings.
5. If the provider throws and stale cache exists, use stale cache with warnings.
6. Otherwise use the existing mock calendar fallback for the current month.

The quote wrapper itself can also return deterministic sample dividends when the route fails. Those are treated as a `sample` provider result and cached as sample-generated calendar events.
