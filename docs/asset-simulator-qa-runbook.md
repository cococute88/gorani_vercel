# 자산 시뮬레이터 QA Runbook

## 개요

### 대상과 준비

- 대상 경로: `/asset-simulator`
- 권장 환경: 최신 Chromium 계열 브라우저, 데스크톱 폭과 모바일 폭 모두 확인
- 저장 확인: 로그인한 계정에서는 클라우드 저장을, 로그아웃 또는 Firebase 미설정 환경에서는 이 브라우저의 로컬 저장을 확인한다.
- 데이터 주의: 포트폴리오 예시는 검증용이며 투자 권유가 아니다. 실시간/외부 데이터 상태에 따라 자동 계산 결과는 달라질 수 있다.

### 확인 원칙

1. 포트폴리오 입력·자동 계산 결과는 **가정 적용** 전까지 기존 projection을 바꾸지 않아야 한다.
2. 적용 후에만 포트폴리오 기반 가정과 요약이 projection 및 은퇴 안전성 분석에 반영되어야 한다.
3. 하락장 결과는 비교 전용이며 기본 projection, 차트, 입력값을 덮어쓰면 안 된다.
4. 저장·새로고침 후에는 저장 시점의 설정과 적용 상태가 동일하게 복원되어야 한다.

## 사용자 주요 흐름

다음 순서를 한 번 끝까지 수행한다.

1. `/asset-simulator`에 접속한다.
2. 기본 입력값과 결과 카드/탭이 정상적으로 보이는지 확인한다.
3. **포트폴리오 설정**에서 `예시 포트폴리오 불러오기`를 누르거나 직접 티커와 비중을 입력한다.
4. 계좌별 `전체 자동 계산`을 실행한다.
5. 데이터가 부족하거나 자동 계산에 실패한 티커는 수동 fallback 값으로 보완한다.
6. 유효성 오류가 없는 상태에서 `포트폴리오 가정 적용`을 누른다.
7. **은퇴 안전성 분석**에서 목표 월생활비를 입력한다.
8. 기본 시나리오와 하락장 시나리오의 안전성 결과를 비교한다.
9. 상단 입력 영역의 `저장`을 누르고 성공 또는 로컬 저장 안내를 확인한다.
10. 페이지를 새로고침하여 포트폴리오 설정, 적용된 가정, 목표 월생활비가 복원되는지 확인한다.

## 서브탭 구조 검증

`/asset-simulator`는 계산기 페이지와 동일한 URL query parameter 기반 서브탭으로 나뉜다.
기본 시뮬레이터 입력·결과는 `기본 시뮬레이터` 탭, 포트폴리오 설정과 은퇴 안전성 분석은
`안정성 체크` 탭에 있다. 비활성 탭은 언마운트하지 않고 `hidden`으로만 숨겨 상태를 보존한다.

다음을 확인한다.

1. `/asset-simulator` 또는 `/asset-simulator?tab=basic`으로 접속하면 `기본 시뮬레이터` 탭이 활성이다.
2. `/asset-simulator?tab=safety`로 직접 접속하면 `안정성 체크` 탭이 활성이다.
3. 알 수 없는 값(예: `?tab=garbage`)은 `기본 시뮬레이터`로 fallback한다.
4. 탭을 클릭하면 URL query가 `?tab=safety`/제거로 바뀌고, 새로고침 후에도 탭이 유지된다.
5. `안정성 체크` 탭에서 예시 포트폴리오를 불러온 뒤 `기본 시뮬레이터`로 이동했다가 돌아와도
   입력·자동 계산 결과가 유지된다(비활성 탭이 언마운트되지 않는다).
6. `안정성 체크` 탭 라벨의 상태 도트는 적용 가정이 없으면 회색, 적용됐지만 config 불일치·stale이면
   호박, 적용·일치·최신이면 초록으로 표시된다.
7. 저장 안내 footer는 라이트/다크 모두 배경·글자 대비가 충분하다.

## 안정성 체크 대시보드 검증

`안정성 체크` 탭은 대시보드형 레이아웃(`SafetyCheckDashboard`)으로 구성된다. 상단 Hero Summary,
좌측 포트폴리오 설정 패널, 우측 KPI·시나리오 비교, 하단 계좌별 상세 순서다. 이 UI는 표시/문구
정리이며 계산 로직·Safety 판정·stress 정책·저장 구조를 바꾸지 않는다.

다음을 확인한다.

1. 최상단 Hero Summary(`SafetyHeroCard`)에 통합 안전성 등급(대형 1개), 한줄 판단, 기본→하락장 요약,
   목표 월생활비 인라인 입력, 가정 적용 상태 배지, 저장 버튼이 보인다. 한줄 판단은 단정적 공포
   표현(파산·실패 확정·보장 등) 없이 부드럽게 표시된다. 통합 등급은 Hero에서만 크게 보인다.
2. KPI 카드 3개(월생활비 충당률 / 최종 실질자산 / 하락장 손상폭)가 보인다. 통합 안전성 등급 KPI는
   Hero와 중복이라 제거됐다. 하락장 손상폭은 기본 최종 실질자산 대비 하락장 최종 실질자산 차이로 표시된다.
   각 KPI 카드에는 표시 보조용 CSS mini bar가 있다(충당률 bullet + 하락장 marker, 손상폭 bar,
   최종 실질자산 기본/하락장 비교 bar). mini bar는 표시 보조이며 새 계산 로직이 아니다.
3. `기본 · 하락장 비교`는 카드 2장이 아니라 **단일 비교표**(`SafetyScenarioCompareTable`)로 표시된다.
   열은 지표 / 기본 시나리오 / 하락장 시나리오 / 변화이고, 행은 통합 등급·월생활비 충당률·자산 보존율·
   최종 실질자산 4개다. 각 행의 변화 열에는 방향 기호(▼/▲)와 숫자가 함께 표시되고, 등급 행을 제외한
   행에는 4px delta bar가 보인다. 등급 행은 `B → D (-28.6점)`처럼 텍스트 중심이다.
   - **변화 색상**: 악화는 기본 amber, 다음 경우만 rose로 승격한다 — 충당률 행 하락장 값 90% 미만,
     보존율/최종자산 행 감소율 25% 이상. 개선/동일은 slate. 색만으로 판단하지 않도록 숫자·문구를 함께 표시한다.
   - **악화 항목 요약**: 표 아래에 `하락장에서 약해진 항목: 월생활비 충당률 -25%p, 자산 보존율 -68%p`처럼
     한 줄로 요약되고, rose 승격 항목은 더 강하게 표시된다. 악화가 없으면 `하락장에서도 핵심 지표가 크게
     약해지지 않았습니다.`로 표시된다.
   - **다음 조정 후보**: warnings/metrics 기반 점검 항목 2~3개가 칩으로 보인다(예: `목표 생활비 조정 검토`,
     `배당 현금흐름 보강 점검`, `현금·안전자산 비중 점검`, `포트폴리오 가정 재확인`). 새 투자 조언이 아니라
     참고용 점검 제안이며 `투자 권유가 아닙니다` 안내가 함께 보인다.
   - **하락장 설명**: 표 상단에 `하락장은 은퇴 초반 하락·첫 3년 저수익·배당 20% 삭감을 가정한 보수적 점검`
     문구가 한 번만 노출되어 중복 배너가 줄었다.
4. 계좌별 상세(절세·위탁·통합)는 `은퇴 안전성 분석` 아래 접기/펼치기(아코디언)에 있으며 기본은 접힘이다.
   이번 비교표 변경은 계좌별 상세 구조를 바꾸지 않는다(PR-3 범위).
5. 데스크톱에서 좌측 설정 패널은 sticky로 동작하고, 모바일(375px)에서는 판단·KPI가 먼저,
   설정 패널이 그 아래, 상세가 접힘으로 배치된다. 375px에서 비교표는 세로 카드형 row로 전환되어
   가로 스크롤(넘침)이 없어야 한다.
6. **평가 기준 표시**: 목표 월생활비가 어느 평가에 반영되는지 오해하지 않도록 기준이 보인다.
   - 비교표 상단(통합): 목표 입력 전 `평가 기준: 임시 인출 기준`, 목표 입력 후 `평가 기준: 목표 월생활비 300만원`.
   - 계좌별 상세 카드: 절세 `평가 기준: 절세계좌 인출 계획 기준`, 위탁 `평가 기준: 배당 현금흐름과 자산 보존성`.
7. 목표 월생활비 입력 전/후로 Hero 입력 안내(`입력하면 목표 생활비 기준으로 평가합니다`↔`목표 300만원 기준으로 평가 중`),
   충당률 KPI(`목표 입력 시 표시`↔`%`), 충당률 mini bar, 통합 평가 기준 문구가 함께 바뀐다.
   목표 미입력 시 충당률 KPI에는 Hero 입력창으로 이동하는 `목표 월생활비 입력하기 →` 버튼이 보인다.
   계좌별(절세/위탁) 기준 문구는 목표 입력과 무관하게 고정이다.
8. 라이트 모드에서 요약 바·KPI·평가 기준 보조문구의 글자 대비가 충분하다(보조문구 `text-slate-600` 이상).
9. 다크 모드에서 카드/배지/평가 기준 박스의 대비가 유지된다.

## 포트폴리오 설정 검증

### 절차

1. 포트폴리오 설정이 비어 있으면 `예시 포트폴리오 불러오기`를 선택한다. 예시는 절세계좌 `SCHD`/`QLD`, 위탁계좌 `SCHD`/`JEPQ`를 제공한다.
2. 직접 입력 시에는 계좌별로 티커와 비중을 입력하고, 필요하면 `+ 행 추가`로 종목을 추가한다.
3. 각 계좌의 비중 합계가 100%인지 확인한다. 티커·비중 오류가 있으면 화면의 검증 메시지를 확인한다.
4. 자동 계산 또는 수동 입력을 마친 뒤 `포트폴리오 가정 적용` 버튼 상태를 확인한다.

### Expected result

- 설정이 없을 때 예시 불러오기와 빈 포트폴리오 시작 선택지가 표시된다.
- 티커는 정규화되어 표시되고, 계좌별 비중 합계와 입력 오류가 즉시 표시된다.
- 비중이 100%가 아니거나 필수 metric이 없으면 적용 버튼이 비활성화되고 원인이 안내된다.
- 자동 계산 결과나 수동 입력만으로는 projection이 바뀌지 않는다.
- 적용에 성공하면 `적용된 가정으로 시뮬레이션 반영됨` 상태와 포트폴리오 요약이 표시된다.
- 적용 후 설정을 수정하면 적용 상태가 변경됨/재적용 필요로 표시되어, 이전 가정을 무심코 계속 사용하지 않게 한다.

## 자동 계산 검증

### 절차

1. 각 계좌에서 자동 계산 모드인 티커를 하나 이상 준비한다.
2. `전체 자동 계산`을 누르고, 요청 중에는 `자동 계산 중…` 상태와 적용 제한이 표시되는지 확인한다.
3. 정상 티커의 metric 상태와 계산 시각을 확인한다.
4. 실패 또는 보완 필요 티커가 있으면 `실패한 티커만 다시 계산`이 표시되는지 확인한다.
5. 모든 필수 metric이 충족된 뒤에만 가정을 적용한다.

### Expected result

- 자동 계산은 티커별로 상태를 표시하고, 정상 결과는 적용 가능한 가정의 재료가 된다.
- 자동 계산 실패 티커는 실패 상태와 원인을 표시하며, 성공한 티커의 결과를 삭제하거나 기본 projection을 변경하지 않는다.
- 진행 중인 자동 계산이 있으면 가정 적용은 가능하지 않다.
- 자동 계산 결과가 오래되었거나 현재 설정과 맞지 않으면 재계산/재적용 안내가 표시된다.

## 수동 fallback 검증

### 절차

1. 자동 계산이 실패하거나 이력 부족 안내가 표시된 티커를 선택한다. 짧은 이력의 ETF(예: `JEPQ`)가 재현 후보가 될 수 있다.
2. 해당 행의 metric 모드를 수동으로 전환한다.
3. 계좌 유형에 맞는 값을 입력한다.
   - 절세계좌: 총수익 CAGR
   - 위탁계좌: 가격 CAGR, 배당률(TTM), 배당성장률
4. 비중 합계와 다른 입력 오류를 정리한 뒤 가정 적용을 시도한다.

### Expected result

- 자동 계산 실패/이력 부족 상태는 수동 fallback이 필요한 상태로 명확히 표시된다.
- 수동 필수 값을 모두 채우면 해당 티커가 적용 가능 상태가 된다.
- 수동 fallback 적용 전에는 projection이 바뀌지 않는다.
- 유효한 수동 값을 포함해 가정을 적용하면 포트폴리오 기반 projection과 안전성 분석이 갱신된다.

## 목표 월생활비 검증

### 절차

1. 상단 Hero Summary의 `목표 월생활비` 입력(`id="target-monthly-expense"`)에 0보다 큰 값을 입력한다.
   입력 단위는 현재 가치 기준 만원이다. (목표 미입력 시 충당률 KPI의 `목표 월생활비 입력하기 →`
   버튼을 누르면 이 입력창으로 포커스가 이동한다.)
2. 기본과 하락장 카드의 충당률·통합 평가 문구를 확인한다.
3. 입력값을 비우고 기존 참고용 평가로 돌아가는지 확인한다.

### Expected result

- 목표 월생활비가 있으면 통합 안전성 평가는 기존 proxy 기준이 아니라 입력한 target 기준으로 전환된다.
- 기본 및 하락장 시나리오 모두 같은 목표 월생활비를 기준으로 충당률을 표시한다.
- 값을 비우면 목표 설정은 `null`로 취급되고, 기존 인출 조건 기반의 참고용 임시 평가가 다시 표시된다.

## 기본/하락장 안전성 비교 검증

### 절차

1. 기본 입력 또는 적용된 포트폴리오 가정으로 결과를 준비한다.
2. 안정성 체크 탭의 `기본 · 하락장 비교` 단일 비교표에서 지표별 기본/하락장/변화를 확인한다.
3. 변화 열의 방향 기호·delta bar·색상, 표 아래 악화 항목 요약과 다음 조정 후보를 확인한다.
4. 기본 결과 탭/차트와 입력값이 비교 전후 동일하게 유지되는지 확인한다.

### Expected result

- 기본 시나리오는 현재 입력과 적용된 가정을 그대로 사용한다.
- 하락장 시나리오는 은퇴 직후 하락, 첫 3년 저수익, 위탁 배당 20% 삭감의 보수적 비교 결과를 표시한다.
- 비교표의 변화 열은 방향 기호(▼/▲)와 숫자를 함께 표시하고, 악화 정도에 따라 amber/rose로 구분한다.
  (충당률 90% 미만, 보존율/최종자산 25% 이상 감소만 rose로 승격한다.)
- 하락장 시나리오는 기본 projection을 덮어쓰지 않는다. 기본 결과 탭·차트·입력값은 비교 전후 동일해야 한다.
- 하락장 표시 점수는 같은 계정의 기본 점수를 넘지 않는다(점수 경쟁이 아니라 손상 정도 점검).
- 두 시나리오는 같은 목표 월생활비가 설정되어 있으면 같은 target으로 평가한다.

## 저장/복원 검증

### 절차

1. 포트폴리오 설정을 만들고 자동 또는 수동 metric을 준비한 뒤 가정을 적용한다.
2. 목표 월생활비를 입력한다.
3. `저장`을 누른다.
4. 로그인 상태면 `저장 완료 · 클라우드에 동기화됨`을, 로그아웃 상태면 로컬 저장 안내를 확인한다.
5. 페이지를 새로고침한다. 로그인 상태에서는 최신 저장본이 로컬/클라우드 중에서 선택되어 복원되는지 확인한다.

### Expected result

- 저장 후 새로고침하면 `portfolioConfig`, `portfolioAssumptions`, `retirementSafetyConfig`가 복원된다.
- 복원된 가정이 적용 상태라면 projection과 은퇴 안전성 분석도 그 가정을 반영한다.
- 목표 월생활비와 기본/하락장 충당률 기준도 저장 전과 동일하다.
- 로그아웃 또는 Firebase 미설정 환경에서는 로컬 저장이 유지되고, 클라우드 동기화 성공으로 표시되지 않는다.
- 클라우드 읽기/쓰기 실패 시 로컬 저장값을 유지하는 안내 또는 오류 메시지가 표시되며, 브라우저가 빈 상태로 초기화되지 않는다.

## 알려진 경고/제한사항

- Recharts `2.12.7`의 내부 `defaultProps` 개발 경고는 애플리케이션 코드 오류가 아니다.
- `metadataBase` 경고는 PR #188에서 해결되었다. 현재 `app/layout.tsx`의 `metadataBase` 설정을 임의로 되돌리지 않는다.
- 브라우저 스크린샷 도구는 Recharts가 많은 페이지에서 시간 초과될 수 있다. 이 경우 화면 수동 확인과 DOM/콘솔 확인으로 분리해 기록한다.
- `JEPQ`처럼 이력이 짧은 ETF는 자동 metric이 부족할 수 있으며, 이때는 수동 fallback 값을 입력해야 한다.
- Firebase가 설정되지 않은 로컬 환경에서는 관련 로그 또는 로컬 미리보기 안내가 나타날 수 있다. 이 환경에서는 클라우드 동기화 성공을 기대하지 않는다.
- 이 결과는 참고용 시뮬레이션이며 투자 권유가 아니다. 자동 데이터의 출처·가용성에 따라 결과가 달라질 수 있다.

## 후속 개선 후보

- 현재 하락장 검증은 계좌 단위 `early_downturn` preset 중심이다.
- holding별 stress, 현금/예비금 stress, 여러 stress preset 선택 기능은 별도 후속 작업으로 검토한다.
- 하락장 시나리오는 기본보다 좋아지는지를 판단하는 지표가 아니라, 동일 조건에서 손상 정도를 확인하는 보수적 점검이다. 비교 카드의 표시 점수는 사용자가 반대로 해석하지 않도록 같은 계정의 기본 점수를 넘지 않는다.
- 자산 보존율이 1,000% 이상이면 UI 카드에서는 `1,000% 이상`으로 상한 표시할 수 있다. 계산에 사용되는 raw metric은 내부 결과에 그대로 유지된다.
- 자동 metric 데이터의 범위·신뢰도 표시와 이력 부족 ETF의 수동 fallback UX를 더 세분화한다.
- 저장/복원 QA를 로그인·로그아웃·Firebase 미설정 환경별 자동화 시나리오로 확장한다.
